# 4-Week Curriculum
## Junior Front-End Sprint with Bootstrap 4.4.1

## Why this version exists

Your review is in the **first week of May 2026**, so this plan is compressed into **4 execution weeks** with a **1-week polish buffer** before the review.

### Timeline
- **Week 1:** March 30-April 5, 2026
- **Week 2:** April 6-April 12, 2026
- **Week 3:** April 13-April 19, 2026
- **Week 4:** April 20-April 26, 2026
- **Buffer / Review Prep:** April 27-May 3, 2026

## Assumptions

- **Time available:** 10-12 focused hours per week
- **Goal:** reach credible junior front-end readiness, not full framework mastery
- **Primary stack:** HTML, CSS, JavaScript, Bootstrap 4.4.1, GitHub, accessibility, deployment
- **Scope rule:** ship fewer things, but make them cleaner and easier to explain
- **Stretch only if time remains:** Datastar / app-specific component annotation work

## Sprint strategy

This is **not** a broad 8-week exploration anymore. It is a **proof-of-readiness sprint**.

### What you will prove by the review
- you can build semantic HTML pages
- you can use Bootstrap 4.4.1 for layout and common UI components
- you can write custom CSS without fighting the framework
- you can build small JavaScript interactions and data-driven UI
- you can use GitHub like a real junior developer: issues, branches, pull requests, evidence, retrospectives
- you can explain your choices clearly to a manager

## Success metrics by April 26

- **2 deployed projects**
- **1 review-ready capstone demo** (either a third mini-project or a polished upgrade of Project 2)
- **8-12 closed issues**
- **4-6 pull requests**
- **4 weekly retrospectives**
- **1 clear GitHub Project board**
- **1 evidence log**
- **1 short manager demo video or walkthrough**
- **1 accessible form**
- **1 responsive multi-breakpoint layout**
- **1 JavaScript app with real state or data handling**

## Owners / roles

| Role | Owner | Responsibility |
|---|---|---|
| Learner / builder | You | build, commit, deploy, document, demo |
| Reviewer / coach | Manager | review progress, ask questions, judge readiness |
| Assistant / study support | ChatGPT | help plan, debug, explain, and review artifacts |

## Manager review package

By the time your review starts, you should be able to hand over:

- repo link
- GitHub Project board
- 2 deployed URLs
- README links
- 1 evidence log

- 1 summary message explaining what you learned in 4 weeks

## Fast-track operating rules

- ship **every week**
- deploy **every Friday**
- no framework hopping
- keep scope small enough to finish
- every meaningful task starts as a GitHub issue
- every visible change goes through a PR
- screenshots and notes are mandatory for UI work

## Risks and mitigations

| Risk | Why it happens | Mitigation |
|---|---|---|
| Overloading the schedule | trying to learn too many tools at once | keep to HTML, CSS, Bootstrap 4.4.1, JavaScript, GitHub |
| Weak demo quality | too much building, not enough polish | reserve the buffer week for demo prep only |
| Projects feel unfinished | scope too large | prefer a smaller finished app over a larger half-built app |
| Bootstrap dependence hides fundamentals | relying only on copied classes | explain each class choice and pair it with semantic HTML |
| Anxiety slows shipping | perfectionism | use weekly exit criteria instead of endless tweaking |

## Review-day talking points

Prepare answers to these:
- What did Bootstrap 4.4.1 save you time on?
- Where did you still need custom CSS?
- What JavaScript did you write yourself?
- How did GitHub help you stay organized?
- What are the next 30 days of improvement after the review?
